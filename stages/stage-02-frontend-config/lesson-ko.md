# Stage 02: 프론트엔드 설정 노출

이 단계는 프론트엔드 설정값이 브라우저에 노출되는 문제를 설명합니다. 여기 있는 값은 모두 가짜이며 실제 서비스에 연결되지 않습니다.

## 서버 전용 환경 변수와 클라이언트 공개 환경 변수의 차이

서버 전용 환경 변수는 서버 코드에서만 읽어야 합니다. 예를 들어 관리자 API 키, 내부 관리자 URL, 디버그 토큰은 서버에서만 사용해야 합니다.

반대로 클라이언트 공개 환경 변수는 브라우저 번들에 포함될 수 있습니다. 사용자는 개발자 도구나 빌드된 JavaScript 파일을 통해 이 값을 볼 수 있습니다.

## 왜 `NEXT_PUBLIC_`, `VITE_`, `REACT_APP_`가 중요한가요?

프레임워크마다 "이 값은 브라우저에 공개해도 된다"는 의미의 접두사가 있습니다.

- Next.js: `NEXT_PUBLIC_`
- Vite: `VITE_`
- Create React App: `REACT_APP_`

이 접두사가 붙은 값은 프론트엔드 코드에서 사용할 수 있게 번들에 들어갈 수 있습니다. 그래서 `NEXT_PUBLIC_ADMIN_API_KEY`, `VITE_BACKEND_ADMIN_TOKEN`, `REACT_APP_DEBUG_TOKEN`처럼 권한이 있어 보이는 값에 이런 접두사를 붙이면 위험합니다.

## 안전한 수정 방법

권한이 필요한 값은 백엔드에만 둡니다. 프론트엔드는 관리자 키나 내부 URL을 직접 받지 않고, 필요한 작업을 백엔드 API에 요청해야 합니다.

예를 들어 프론트엔드가 `adminApiKey`를 들고 관리자 API를 직접 호출하는 대신:

1. 프론트엔드가 `/api/projects` 같은 일반 백엔드 route를 호출합니다.
2. 백엔드가 현재 사용자의 권한을 확인합니다.
3. 백엔드가 서버 전용 환경 변수로 필요한 내부 작업을 수행합니다.
4. 백엔드는 브라우저에 필요한 결과만 돌려줍니다.

이렇게 하면 브라우저에는 권한 있는 키나 내부 관리자 URL이 전달되지 않습니다.
