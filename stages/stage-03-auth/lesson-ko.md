# Stage 03: 약한 인증과 권한 경계

이 단계는 "로그인은 된 것처럼 보이지만 서버가 실제로 믿을 수 있는 검사를 하지 않는" 문제를 보여줍니다. 예제는 모두 로컬 교육용이며, 토큰과 비밀번호는 가짜 값입니다.

## 인증과 인가의 차이

인증(authentication)은 "사용자가 누구인지 확인하는 과정"입니다. 예를 들어 이메일과 비밀번호로 Alice인지 확인하는 것이 인증입니다.

인가(authorization)는 "그 사용자가 이 행동을 해도 되는지 확인하는 과정"입니다. Alice로 로그인했더라도 관리자 사용자 목록을 볼 권한은 없을 수 있습니다.

좋은 서버는 두 가지를 모두 확인합니다. 먼저 신뢰할 수 있는 세션으로 사용자를 확인하고, 그 다음 데이터베이스나 서버 세션에 저장된 역할로 권한을 확인합니다.

## 클라이언트가 보낸 역할을 믿으면 안 되는 이유

Stage 03의 `/api/admin/users` 예제는 `req.body.isAdmin === true`를 믿습니다. 사용자는 브라우저 개발자 도구나 curl로 요청 본문을 마음대로 바꿀 수 있습니다. 그래서 `isAdmin: true` 같은 값은 권한 근거가 될 수 없습니다.

안전한 방식은 서버가 세션에서 사용자 ID를 읽고, 서버가 가진 사용자 데이터에서 역할을 조회하는 것입니다.

```ts
const user = await loadUserFromTrustedSession(req);
if (user.role !== "admin") {
  return res.status(403).json({ error: "admin only" });
}
```

## 토큰은 디코드만 하면 안 됩니다

JWT는 보통 헤더, 페이로드, 서명으로 구성됩니다. 페이로드를 디코드하는 것은 내용을 읽는 것일 뿐입니다. 서명을 검증하지 않으면 누군가 페이로드를 바꿔도 서버가 알아차리지 못할 수 있습니다.

안전한 방식은 서명, 만료 시간, 발급자, 대상 값을 모두 검증한 뒤에만 토큰 안의 `sub`나 `role`을 신뢰하는 것입니다.

## 로그인에는 속도 제한이 필요합니다

로그인 API에 속도 제한이 없으면 같은 계정에 대해 비밀번호를 계속 추측할 수 있습니다. 실제 서비스에서는 Redis 같은 저장소로 IP, 계정, 시간 구간별 실패 횟수를 기록하고 일정 횟수 이상이면 잠시 막아야 합니다.

## 비밀번호 재설정 토큰에는 만료 시간이 필요합니다

재설정 토큰은 짧게 살아야 합니다. 토큰을 만들 때 `expiresAt`을 저장하고, 사용할 때 현재 시간이 만료 시간을 넘었는지 확인해야 합니다. 사용한 토큰은 바로 삭제하는 것이 좋습니다.

## 안전한 수정 방향

- 역할은 요청 본문이 아니라 서버 세션이나 데이터베이스에서 읽습니다.
- API 라우트 내부에서 권한을 다시 확인합니다.
- JWT는 디코드가 아니라 검증을 먼저 합니다.
- 로그인에는 Redis 기반 rate limit을 둡니다.
- 재설정 토큰에는 `expiresAt`과 1회 사용 정책을 둡니다.
