# Stage 06: 사용자 콘텐츠와 XSS 위험

이 단계는 사용자가 입력한 내용이나 AI가 만든 텍스트를 HTML로 렌더링할 때 생기는 위험을 보여줍니다. 예제 문자열은 모두 무해한 교육용 서식 문자열입니다.

## 텍스트 렌더링과 HTML 렌더링의 차이

텍스트 렌더링은 사용자가 입력한 값을 글자로 보여주는 것입니다.

```tsx
<p>{user.bio}</p>
```

React는 이런 텍스트를 기본적으로 escape합니다. 즉 `<strong>` 같은 문자열도 HTML 태그로 실행하지 않고 글자로 보여줍니다.

HTML 렌더링은 문자열을 브라우저가 HTML로 해석하게 만드는 것입니다.

```tsx
<div dangerouslySetInnerHTML={{ __html: user.bio }} />
```

이 방식은 정말 필요한 경우에만 사용해야 하며, 사용자 입력을 그대로 넣으면 위험합니다.

## Stored XSS가 위험한 이유

Stored XSS는 사용자가 저장한 콘텐츠가 나중에 다른 사람의 브라우저에서 HTML로 렌더링되는 문제입니다. 예를 들어 프로필 bio, 문의 메시지, AI가 만든 소개글이 데이터베이스에 저장되고, 다른 사용자가 그 페이지를 열 때 실행될 수 있습니다.

## 관리자 페이지는 더 위험한 sink입니다

관리자 페이지는 권한이 높은 사람이 보는 화면입니다. 문의 메시지나 신고 내용처럼 외부 사용자가 제출한 콘텐츠를 관리자 화면에서 HTML로 렌더링하면, 관리자 권한이 있는 브라우저가 위험한 콘텐츠를 처리하게 됩니다.

## Sanitization이란 무엇인가요?

Sanitization은 HTML을 완전히 금지하지는 않더라도 위험한 태그와 속성을 제거하는 과정입니다. 예를 들어 굵게 표시하는 `<strong>`은 허용하고, 위험한 스크립트 실행 가능성은 제거하는 식입니다.

하지만 직접 문자열 치환으로 sanitizer를 만들면 빠뜨리기 쉽습니다. 실제 프로젝트에서는 검증된 sanitizer 라이브러리를 사용하고, 어떤 태그와 속성을 허용할지 명확히 정해야 합니다.

## CSP는 보조 방어입니다

Content Security Policy(CSP)는 브라우저에게 어떤 스크립트와 리소스를 허용할지 알려주는 보안 헤더입니다. 좋은 CSP는 XSS가 발생했을 때 피해를 줄이는 데 도움이 됩니다.

하지만 CSP는 sanitization을 대신할 수 없습니다. 먼저 사용자 입력을 안전하게 렌더링하고, 그 다음 CSP를 defense-in-depth로 사용해야 합니다. `unsafe-inline`, `unsafe-eval`, 넓은 `default-src *` 같은 설정은 방어 효과를 크게 낮춥니다.

## 안전한 수정 방향

- 사용자 bio와 문의 메시지는 JSX 텍스트로 렌더링합니다.
- Markdown에서 raw HTML을 비활성화하거나, 렌더링 결과를 sanitizer에 통과시킵니다.
- 관리자 페이지에서는 제출된 콘텐츠를 특히 조심합니다.
- CSP는 엄격하게 설정하되, sanitization의 대체재로 생각하지 않습니다.
