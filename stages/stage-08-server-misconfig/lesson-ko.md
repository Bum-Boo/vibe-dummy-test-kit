# Stage 08: 서버와 인프라 설정 실수

프론트엔드와 백엔드 코드가 괜찮아 보여도 서버 설정이 약하면 보안 문제가 생길 수 있습니다. Docker, Nginx, CORS, 로그, public 디렉터리 같은 주변 설정도 애플리케이션 보안의 일부입니다.

## 컨테이너를 root로 실행하면 왜 위험한가요?

컨테이너 안에서 프로세스를 root로 실행하면, 앱이 침해되었을 때 컨테이너 내부에서 더 큰 권한을 갖게 됩니다. 안전한 방식은 앱 전용 사용자를 만들고 `USER app`처럼 낮은 권한으로 실행하는 것입니다.

## `.env`를 이미지에 복사하면 왜 위험한가요?

`.env` 파일은 토큰, 비밀번호, 내부 설정을 담는 경우가 많습니다. 이 파일을 Docker 이미지에 `COPY .env /app/.env`로 넣으면 이미지 레이어, 캐시, 배포 산출물에 값이 남을 수 있습니다. 이 실습에서는 모두 가짜 값이지만 실제 프로젝트에서는 절대 넣으면 안 됩니다.

## CORS가 하는 일과 하지 않는 일

CORS는 브라우저가 다른 origin으로 요청할 수 있는지를 제한하는 브라우저 보안 정책입니다. 서버 간 요청을 막는 방화벽이 아니고, 인증이나 권한 검사를 대신하지도 않습니다.

민감한 API에 `origin: "*"`를 설정하면 어떤 웹사이트에서든 브라우저 요청을 시도할 수 있게 됩니다. 실제 서비스에서는 신뢰하는 origin만 허용해야 합니다.

## 로그도 정보 유출 경로입니다

로그는 개발자, 운영 도구, 파일, 외부 로그 수집기로 이동할 수 있습니다. `Authorization` 헤더나 session token을 그대로 로그에 남기면 토큰이 여러 곳에 복사됩니다.

안전한 로그는 토큰 값을 남기지 않고, 필요하다면 "헤더가 있었는지" 정도만 기록합니다.

## public 백업 파일이 위험한 이유

`public/backups/users-backup.json`처럼 public 디렉터리에 파일을 두면 브라우저에서 직접 접근할 수 있습니다. 앱 라우트가 아무리 안전해도 public 파일은 별도의 권한 검사 없이 제공될 수 있습니다.

백업은 web root 밖에 저장하고, 다운로드가 필요하면 서버 라우트가 인증과 권한을 확인한 뒤 제공해야 합니다.

## 안전한 방향

- 컨테이너는 non-root 사용자로 실행합니다.
- `.env`는 `.dockerignore`로 제외하고 런타임에만 주입합니다.
- Nginx나 앱에서 CSP, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy` 같은 보안 헤더를 설정합니다.
- CORS는 신뢰하는 local origin만 허용합니다.
- debug endpoint는 기본적으로 비활성화합니다.
- 로그에서 인증 헤더와 세션 토큰을 제거합니다.
- 백업 파일은 public 디렉터리 밖에 둡니다.
