# Stage 09: AI 기능 보안 위험

이 단계는 AI 기능을 빠르게 붙일 때 생기는 보안 실수를 보여줍니다. 모든 응답은 로컬 mock LLM이 만든 고정 문자열이며, 실제 AI API나 실제 API 키를 사용하지 않습니다.

## AI 출력은 신뢰할 수 없는 입력입니다

AI가 만든 텍스트는 사용자가 입력한 값과 비슷하게 취급해야 합니다. 모델이 만든 HTML, SQL, 명령어, 요약문은 "서버가 만든 안전한 값"이 아닙니다. 화면에 보여주거나 실행하기 전에 반드시 검증해야 합니다.

## 생성된 SQL, 코드, 명령어를 바로 실행하면 안 됩니다

취약한 예:

```ts
const generatedSql = mockLlm(...).text;
return executeGeneratedSql(generatedSql);
```

모델이 만든 SQL을 그대로 실행하면 모델이 데이터베이스 동작을 결정하게 됩니다. 명령어도 마찬가지입니다. 이 실습에서는 명령어 실행을 하지 않고 dry-run 계획만 반환하지만, 실제 앱에서 `exec` 같은 함수에 연결하면 매우 위험합니다.

안전한 방식은 사용자가 선택할 수 있는 동작을 서버가 미리 정해두는 것입니다.

```ts
safeSqlAssistant("free-users");
safeCommandHelper("list-demo-files");
```

## 프롬프트 안의 secret은 secret이 아닙니다

시스템 프롬프트에 내부 토큰이나 비밀 값을 넣어도 안전하지 않습니다. 프롬프트는 코드 저장소, 로그, 디버그 화면, 모델 응답을 통해 노출될 수 있습니다. 이 단계의 `fake_internal_ai_prompt_secret_for_training_only`는 가짜 교육용 문자열입니다. 실제 값은 절대 프롬프트에 넣으면 안 됩니다.

## 입력 길이 제한과 rate limit이 필요한 이유

AI 기능은 비용과 지연 시간이 큽니다. 입력 길이 제한이 없으면 매우 긴 문서를 보내 비용을 늘리거나 처리를 느리게 만들 수 있습니다. rate limit이나 cost guard가 없으면 반복 호출로 리소스를 낭비할 수 있습니다.

## RAG 문서도 신뢰할 수 없습니다

RAG는 문서를 검색해 프롬프트에 넣는 방식입니다. 하지만 검색된 문서 안에 "이전 지시를 무시하라" 같은 문장이 있을 수 있습니다. 문서는 지시문이 아니라 데이터로 다뤄야 합니다.

안전한 방향:

- 시스템 지시와 문서 내용을 명확히 분리합니다.
- 문서를 인용된 데이터로 넣고, 모델이 문서 내용을 명령처럼 따르지 않게 합니다.
- 모델 출력이 할 수 있는 작업을 제한합니다.

## 안전한 대안

- HTML은 템플릿과 escape를 사용합니다.
- SQL은 allowlist된 report ID나 query builder를 사용합니다.
- 명령어는 모델이 직접 만들게 하지 말고 서버의 제한된 action 이름으로 매핑합니다.
- 긴 입력은 자르고, 사용자/IP별 rate limit을 둡니다.
- 위험한 작업은 사람이 검토하는 단계를 둡니다.
